## 原始貼文
這是一個完美的起跑點！既然您已經完成了 SDD（軟體設計規格書），且偏好「看著它一步步改」的精密掌控模式，您現在就像是一位手握精準藍圖的總工程師。
以下為您規劃如何帶領 Claude Code 按照您的 SDD 穩健推進，並確保每一步都不失控：
## 🚀 第一步：準備好「進度追蹤錨點」
在您的專案根目錄下，建立一個簡單的文字檔 progress.md（如果 SDD 裡沒有內建任務清單的話）。這能讓您和 Claude 永遠保持在同一個進度畫面上：

# 開發進度- [ ] 任務一：[從 SDD 拆出的第一個功能]
- [ ] 任務二：[從 SDD 拆出的第二個功能]

## 🛠️ 第二步：與 Claude CLI 的實戰過招對話
請在您的終端機啟動 claude，並將現有的 SDD 檔案直接餵給它（假設您的檔案叫 sdd.md 或 spec.md）：
## 1. 建立全局共識（Plan 階段）
您可以這樣對 Claude 說：

「請讀取 @sdd.md 與 @progress.md。我是這個專案的架構師，我需要你扮演高執行力的資深工程師。請先分析這份 SDD，並在不改動任何檔案的前提下（Plan 模式），告訴我你預計如何實作第一個任務？會動到哪些現有的檔案？」

🎯 此時您的主控點：檢查它理解的架構是否與您想的一致。如果它誤解了某個 API 設計，立刻在這裡糾正它。
## 2. 小步快跑實作（Execute 階段）
當您確認計畫可行後，授權它動手，但限制它的 scope（範圍）：
您可以這樣對 Claude 說：

「計畫沒問題。請切換到 Execute 模式，『只』幫我實作第一個任務。完成後，請自動在終端機幫我跑測試（或編譯檢查），並停下來等待我的審查，先不要往下做第二個任務。」

🎯 此時您的主控點：Claude Code 在修改程式碼時，CLI 會跳出 diff（修改對比）。您可以一行行看它寫得好不好，按下 y 允許寫入，或 n 叫它退回修改。
## 3. 驗收並更新狀態（Evaluate 階段）
當它停下來，且測試全過（綠燈）後：
您可以這樣對 Claude 說：

「很好，程式碼品質符合預期。請幫我在 @progress.md 中將第一個任務打勾 [x]，並針對下一個任務提出你的 Plan。」

------------------------------
## 💡 專屬於「看著它改」的進階心法

   1. 善用 @ 符號限制記憶：在 CLI 中對話時，多使用 @sdd.md 提及檔案。這會強迫 Claude 每次都精準對照您的規格，防止它憑空亂造代碼。
   2. 隨時下達 /config 調整權限：如果您希望連它自動跑測試、自動存檔都要經過您按 y/n 确认，可以檢查 Claude Code 的權限設定，拉高您的掌控度。

既然您的 SDD 已經大致完成，為了讓這個工作流能無縫套用到您的專案中：

* 您的 SDD 檔案目前是用什麼格式儲存的（例如：Markdown 檔案、Notion 頁面複製出來的文字，還是 JSON 格式）？
* 您的專案預計要運行的本地測試或編譯指令是什麼（例如：npm run test、pytest 或 go test），好讓我可以幫您把指令直接融合進提示詞中？
