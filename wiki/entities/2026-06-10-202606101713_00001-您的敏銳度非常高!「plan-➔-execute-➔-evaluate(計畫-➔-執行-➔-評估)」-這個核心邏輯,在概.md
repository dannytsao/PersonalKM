## 原始貼文
您的敏銳度非常高！「Plan ➔ Execute ➔ Evaluate（計畫 ➔ 執行 ➔ 評估）」 這個核心邏輯，在概念上確實就是 Ralph Loop（或常說的 Agent 迴圈）的本質。 [1] 
不過在實務操作和工具設計上，Claude Code 內建的 /plan 模式與真正的 Ralph Loop 還是有一些關鍵的「行為差異」。
我們可以把這兩者理解為：

* Claude Code 的內建模式：是「人機協同、點對點」的控制迴圈。
* 真正的 Ralph Loop：是「完全放養、自動化」的持久迴圈。 [2, 3, 4] 

以下為您詳細拆解它們的本質關聯與差別：
------------------------------
## 1. 概念上的共通點（為什麼您會覺得像）
不論是 Claude 內建的流程還是 Ralph Loop，它們都打破了傳統「一句提示詞、一個回應」的單向模式，而是遵循 OODA 循環（觀察、調整、決定、行動）：

* Plan（計畫）：對照需求（PRD/SDD），分析程式庫（Codebase），拆解出接下來要改動的檔案與步驟。
* Execute（執行）：實際動手改代碼、新增檔案。
* Evaluate（評估/驗證）：執行測試指令（如 npm test）、檢查 Linter，確認改動是否正確。 [1, 5, 6, 7] 

------------------------------
## 2. 實務上的核心差異：誰來按「下一步」？
雖然兩者的哲學一樣，但自動化程度和上下文（Context）的處理方式有很大的不同： [3, 8] 
## 💡 Claude Code 的 /plan 與 Execute 模式（人機協同）
在 Claude Code 中，您按下 Shift+Tab 切換到 Plan Mode，它會幫您看扣（Code）、畫架構藍圖、提問，但此時它被限制為「唯讀」，絕對不會改動您的檔案。 [5, 6] 

* 人類的角色：當您看完它的 Plan 覺得可以，您必須手動切換回 Execute Mode，對它說「好，照這個計畫做」，它才會開始動手。
* 痛點：如果它在 Execute 階段跑測試失敗了，它會停下來問您該怎麼辦，或者您得一直坐在電腦前看著它。
* 上下文：所有的對話紀錄（聊天歷史）會一直累積在同一個 Session 裡。 [2, 6, 8, 9] 

## 🤖 真正的 Ralph Loop（全自動放養）
Ralph Loop 的本質是用一個外部的程式腳本（例如 Bash 的 while 迴圈）去強行包裝 AI 代理。 [4, 8] 

* 運作方式：
1. 腳本啟動 Claude Code 執行 Plan。
   2. Claude 執行 Execute 修改檔案，並自動執行測試進行 Evaluate。
   3. 關鍵點：不論成功或失敗，Claude 的這個 Session 會立刻結束（Exit）。
   4. 外部腳本偵測到它結束了，如果 progress.txt 還沒勾選完，腳本會立刻重啟一個全新、乾淨的 Claude Session，把更新後的進度丟給它，逼它繼續跑下一個循環。 [1, 4, 8, 10] 
* 人類的角色：啟動腳本後，人類就可以去睡覺或喝咖啡了。 [3, 11] 

------------------------------
## 3. 對比總覽表

| 特性 [1, 2, 3, 4, 8, 10, 12, 13, 14] | Claude Code 內建切換 (/plan) | 外部 Ralph Loop 實作 |
|---|---|---|
| 自動化程度 | 半自動。每完成一個階段，通常需要人類確認或下指令推進。 | 全自動。不撞南牆不回頭，直到測試全過或任務全部勾選完成。 |
| 中斷機制 | 遇到編譯錯誤或測驗失敗時，會停下來發問。 | 遇到失敗時，會當成錯誤訊息直接餵給下一次循環，自主嘗試修復。 |
| Context（記憶） | 對話會愈拉愈長，後期可能因為 Token 太多而產生記憶糢糊。 | 每次循環都是全新、乾淨的視窗，只靠外部檔案（如 progress.txt）維持記憶。 |
| 適合場景 | 需要高度謹慎、架構複雜、涉及重大決策的核心開發。 | 大批量、規律性強、有明確測試腳本可以暴力破解的苦工（如修 20 個小 Bug）。 |

## 🎯 結論
您可以說：Claude Code 的 Plan -> Execute -> Evaluate 提供了 Ralph Loop 所需的所有基本原子能力（工具）；而 Ralph Loop 則是把這些原子能力用一個外部的 while 迴圈串聯起來，做成了一條「無人值守的自動化工廠生產線」。 [4, 7] 
在實際開發中，業界最推崇的作法是：
短小的任務，直接在 Claude Code 用 /plan 想清楚後直接做；當發現這是一個要改動幾十個檔案、需要反覆跑測試微調的大工程時，再把專案交給 Ralph Loop 腳本 讓它去背景跑。 [3, 4, 5] 
您目前在開發上，是比較偏好「看著它一步步改（Claude Code 內建模式）」，還是希望「設定好規格就讓它自己去跑（Ralph Loop）」呢？
