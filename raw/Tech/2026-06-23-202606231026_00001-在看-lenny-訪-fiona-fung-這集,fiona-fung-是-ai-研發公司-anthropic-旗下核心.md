## 原始貼文
在看 Lenny 訪 Fiona Fung 這集，Fiona Fung 是 AI 研發公司 Anthropic 旗下核心產品 Claude Code 與 Claude Cowork 的產品暨工程負責人（Leader for Engineering and Product），做RD領域 25 年，在 Meta 帶過五百人。節目開頭提到一個數字，Anthropic 工程師現在每季產出的 code 是四年前的八倍。Fiona分享產出變多之後，瓶頸換去哪了。她的答案是驗證跟品質。
另外訪談中也分享她自己的管理做法tips。早上設一個 routine 幫她把回饋掃完、歸納問題、能修的先修好等她看，醒來就直接有東西可以決策。講到招人只找兩種，有產品 sense 的 builder，跟非常有系統底蘊的專家。衡量團隊她不太看 token 用多少，比較在意這些動作有沒有變成成果，她分享別把動作誤認為進步。另外訪談中Lenny也有問到關於他們新主管進來一律先當IC (Individual Contributor 個人貢獻者)，再開始管人的制度原因，也分享最擁抱ai的工程師差距最大的特質是在成長心態及面對恐懼的方式，還有現在產品快速迭代，規劃時間越來越短，現在是JIT 月度規劃，從原本六個月路線圖到即時規劃

▍ Fiona 的背景：25 年工程師生涯
Lenny 說他在大約一個月前參加了 Code with Claude 活動，聽了 Fiona 的演講之後就想著一定要邀她上 podcast，因為她的思維遠遠超前了大多數人正在思考的方向。在加入 Anthropic 之前，Fiona 在 Microsoft 帶過打造 TypeScript 和 Visual Studio 的團隊。之後她去了 Facebook，從零開始建立 Facebook Marketplace，從點子到上線一手推動，如今 Facebook Marketplace 每年創造超過一千億美元的 GMV。她也參與了 Meta 第一款智慧眼鏡和 Orion AR 眼鏡的開發。後來她到 Instagram 負責基礎架構、成長、內容審核與安全團隊，在 Meta 期間管理超過五百人的組織。她擁有超過 25 年的工程經驗。
Fiona 回顧自己在 IBM 實習時期用 Vim 寫程式、沒有 IDE 的日子。當她加入 Microsoft 被分配到 Visual Studio 團隊時，她甚至不知道 IDE 是什麼，還以為 Visual Studio 是比小畫家更好的繪圖軟體，結果主管臉上的表情讓她印象深刻。但 Visual Studio 最終成為她職涯前 11 年的熱愛所在。她在 VS 編輯器團隊用 VS 編輯器來開發 VS 編輯器本身，這就是她愛上 dogfooding 的起點。在那個社群媒體還沒出現的年代，團隊成員彼此就是最快速的回饋來源，因為大家都是 VS 的重度使用者。
她也提到那個年代軟體是燒在光碟上出貨的，截止日期硬到不行，因為你得確保軟體能交付給工廠壓片、然後上架販售。這就是為什麼那時候會投入大量時間做規劃，因為工程時間是極其珍貴的資源。而現在的轉變是，寫程式不再是瓶頸了。
Fiona 特別提到她非常感恩能拿到 Microsoft 的 offer，因為 2000 年網路泡沫破裂，她那屆畢業生很多公司都不招人或凍結招聘。加入 Visual Studio 團隊讓她第一次接觸到 IDE 的世界，看到斷點調試、多執行緒除錯這些功能時，她的世界觀徹底被改變了。從 IBM 到 Visual Studio，從 Vim 到 IDE，這是她經歷的第一次階梯式巨變。

▍ 8 倍程式碼產出：寫程式不再是瓶頸
Anthropic 發了一則推文顯示工程師每季產出的程式碼量是 2021 到 2025 年間的八倍。Lenny 說那張圖表就是穩定穩定穩定穩定，然後突然直線飆升，像火箭一樣射向月球。工程師這個角色的變化速度令人難以置信，不久前 100% 的程式碼都是人類寫的，如今正在走向 100% 由 AI 撰寫，如同 Boris 那句名言「coding is salt」。
不只是更多人在提交程式碼，而是不同職能的人也在提交，設計師、PM，Claude Code 團隊裡所有人都在 check in code。當不只產出量暴增、提交者的角色也更多元，要怎麼思考驗證？這就是 Fiona 看到的另一個典範轉移。以前工程時間是最珍貴的資源，有硬性的截止日期（例如軟體要壓光碟上架），所以你會投入大量時間做規劃，確保有限時間內產出最大價值。現在瓶頸已經不在寫程式了，轉移到了驗證和品質。

▍ 2026 年的 AI 化軟體團隊長什麼樣
Fiona 說因為角色界線正在模糊，每個人都開始變成建造者（builder）。她自己做的另一個轉變是，開了一個 Claude Code remote session，掛進團隊所有的 repo 裡面。這樣她就能完整看到每個人正在做的工作。這個 session 同時也能存取所有的 Slack 頻道和追蹤指標。每個月她會跟團隊一起回顧，分享螢幕做一場 Claude Code session，一起看重點項目是什麼、出了哪些產品、市場反應如何、回饋管道裡有什麼訊號。
她說以前她可能只用這些 session 來產生 PR 和修 bug，但現在她用這些 session 來支持她跟團隊成員之間的對話。不只看有沒有出貨，還要看出貨之後市場反應如何，有沒有造成什麼 bug。她有句話是「犯新的錯」，犯錯沒關係，但要犯新的錯，這樣才代表我們一直在學習。如果目標是零失誤，那大概代表你走得不夠快或太過謹慎。
透過 Claude，她可以看跨團隊的 incident，歸納出主題，找出哪些地方可能有品質缺口需要投資。這在一年前是不可能做到的，整個過程太手動了。而且以前大家出貨量沒那麼大，你可以用條列清單就搞定。現在八倍的產出量，這幾乎是唯一能跟上的方式。
Lenny 請 Fiona 回答一個大問題：2026 年的 AI 化軟體團隊到底長什麼樣？Fiona 說最大的轉變是角色正在融合，不再有那麼清楚的職能界線。每個人都在往「builder」的方向靠攏。她自己也做了一個具體的改變，就是用一個常駐的 Claude Code remote session 掛進所有 repo，讓她對團隊所有人的工作都有完整的能見度。這也跟她用 Claude 做管理工作直接相關。

▍ 用 Claude 來做管理
Fiona 的晨間儀式過去是泡杯咖啡，然後瀏覽回饋管道，看有沒有什麼她能幫上忙或補上的缺口。但在 routines 功能推出之後，一切都變了。她現在設定一個 routine 每天早上固定時間執行，自動掃描回饋管道、歸納主題，甚至產生可以直接 review 的 PR。她醒來的時候，已經有一堆準備好的 PR 等她看，而不是以前那種「好，我看完資訊了，接下來我要怎麼處理？」的狀態。
這些回饋來自內部各種管道、Email、社群媒體。團隊成員收到的任何回饋，不管是朋友傳訊息、LinkedIn 或社群上的留言，都會彙整到 Slack 裡。她需要 Claude 的幫助才能跟上這些海量的回饋。
作為管理者，她把日常需要做的事情寫成 routine，讓 Claude 幫她檢查專案進度、找出落後的項目、看誰需要支援、處理一些 polish 修改。Claude 會呈現結果讓她 review，甚至在驗證品質夠好的情況下直接執行。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▍ Code Review 的演進
Fiona 說瘋狂的是，去年他們連 Claude Code Reviews 都還沒有。人工 reviewer 曾經是一個非常大的瓶頸。對於需要深度專業知識的關鍵領域，他們仍然確保有適當的人工審核。但她發現真正有效的是，盡可能把「什麼叫做好」的框架自動化。把 spec check 進 repo，確保 spec 跟程式碼同步更新，Claude Code Review 就能驗證程式碼是否符合你設定的標準。這基本上是測試驅動開發（TDD）的進化版。
Fiona 進一步解釋這跟 TDD（測試驅動開發）的精神相通但做法進化了。她記得 TDD 在 2000 年代很流行，先寫測試再寫程式碼，原則上很好，但感覺像是先吃花椰菜一樣痛苦，因為她最享受的是出貨(ship)和做產品的那種快感。她在 Claude Code 上修的第一個 bug，就是請 Claude 幫她做 TDD，先寫測試、確認測試失敗、然後做修復、測試通過。過去那個寫測試的「稅」，現在已經被自動化了。你甚至可以重新檢視那些存在已久的原則，因為有模型幫你做更多工作，這些原則可能變得更有效率。

▍ 招募：創意建造者與深度系統專家
Fiona 在招募上尋找兩種人才：有產品敏感度的創意建造者（creative builders with product sense），以及負責困難核心的深度系統專家（deep systems experts）。
當她剛加入 Claude Code 團隊時，團隊有很好的產品通才，但她發現缺少具有系統和分散式系統背景的人。她的原則是 trust but verify，模型很強但很多領域仍需驗證，哪裡需要深度專業知識，就值得投資在那裡。另一種人是像做夢的人，他們對產品充滿熱情，有想法就去做，持續看回饋、迭代、打磨，把產品端到端地擁有起來，確保體驗是令人愉悅的。
Lenny 提到「野心」這個詞。他說他認識一個 10X 工程師，以前聽到一個功能想法會說「不行，那太難太複雜了」，但現在會說「不不不，完全可以，我讓 Claude Code 去做就好了」。整個思維轉變了，現在的問題是你的野心能有多大。Fiona 分享了一個工程師的故事，這位工程師本來不是 mobile 工程師，但團隊需要把一個功能擴展到行動端，他說有了 Claude，他現在也能做 Android 開發了。AI 拉高了每個人能做到的天花板。

▍ 成長心態：面對恐懼的方式
什麼樣的人在這個時代做得好？Fiona 說成長心態（growth mindset）真的非常重要。她是在從 Microsoft 轉到 Meta 的時候第一次深刻體會到這件事。過去讓你成功的方式，可能不再適用了，但這很可怕，因為你會想「我一直以來都是靠這些成功的，你現在要我改？」保持好奇心，永遠願意學習。
她也看到一些人的挫折感其實來自恐懼。她的建議是正面迎上，問自己「有什麼是我能做的？什麼在我的掌控範圍內？」因為如果你覺得一切都在你的掌控之外、一切都在發生「在你身上」，那確實非常令人沮喪。換個角度想，它是不是在「為你」發生？有沒有什麼行動是你能採取的？
她分享了自己高中時想當視覺藝術家、家裡買不起電腦的故事。她在加拿大長大，擔心付不起大學學費，但她問自己「我能做什麼？」然後看到加拿大國家銀行在學校貼的徵才海報，去做了銀行櫃員。雖然是最低薪資，但她整個暑假存錢，後來週末也繼續做，靠這筆收入撐過了大學。這個在她掌控範圍內的小行動，改變了她的人生軌跡。
她最愛的兩句話是：「如果你不害怕，你會做什麼？」以及「偶爾做一件讓你害怕的事。」當你在專業領域達到巔峰效率的時候，成長的方式就是去做一件你以前不敢做的事，你會有一段學習曲線下滑期，但這就是你持續推動自己成長的方式。
Lenny 說他在 podcast 上引用最多的一句話就是：「你害怕進入的那個洞穴，藏著你追尋的寶藏。」（The cave you fear to enter holds the treasure you seek.）這是 Joseph Campbell 的名言。Fiona 說她要偷走這句話。Lenny 補充說「不要做所有嚇人的事，像是跳懸崖就不用了，但在職涯決策上，最嚇人的那個選項，往往就是你應該走的方向。」

▍ 恐懼與抗拒：擁抱 AI 與被拋下的人
Lenny 提到一個正在形成的鴻溝，有些人全力擁抱 AI 做得風生水起，有些人則完全沒有跟上，處於被落下的危險中。Fiona 對這個話題非常有熱情，因為這跟她的成長背景有關。
她從香港出生，移民到加拿大時不會說英文，父母都要工作，所以全世界最好的奶奶跟著她一起來照顧她。奶奶也不會英文，在一個陌生國家非常孤立。直到有一個夏天，她們找到一家小毛線店，老闆娘也說廣東話，奶奶在那裡找到了她的編織社群，Fiona 則學會了 macrame（結繩編織）。這個小生意創造了一個溫暖的社群感，也是她對小型企業產生熱情的起點。
後來她在用 Cowork 處理自己的差旅報帳時，覺得太神奇了，那些她不想做的事 Cowork 都幫她做了。她想到她認識的那些小型企業主，他們拼命工作、利潤很薄，經常坐在吧台前面對著一疊帳單做發票和報帳。如果 Cowork 能幫她處理差旅報帳，對他們來說一定更有幫助。
她開始幫幾個朋友導入使用。她承認一開始有點尷尬，因為她平常不太跟朋友聊工作，要開口說「欸我在 AI 公司上班，要不要讓我示範一下？」感覺很不自然。但最後大家都玩得很開心。
幫朋友們 onboarding 的過程也很有收穫，她從他們使用產品的方式中發現了很好的 bug，是真正的雙贏。一個開兩家餐廳的朋友用 Claude 找到下載資料夾深處的菜單（她說那個資料夾就像一個雜物抽屜），然後用一種 Fiona 沒想到的方式來用：比較同區域類似菜系的定價，確保自己的價格對本地居民和觀光客都合理。Claude 回傳了一份很酷的市場分析，朋友還說「我最近去了西雅圖那家餐廳，真的不錯。」每次互動 Fiona 都學到新東西。

▍ 潛在需求（Latent Demand）
Lenny 問 Anthropic 為什麼總能比其他人更早發現巨大的機會，例如 coding 作為使用場景、Cowork 切入知識工作、模型個性的重視。Fiona 說她沒在其他實驗室工作過所以不好比較，但她分享了自己的觀察。Claude Code 團隊和 Cowork 團隊持續關注潛在需求（latent demand），這個詞在 Lenny 的 podcast 上已經出現過好幾次了，可能代表著某種重要的模式。
以 coding 為例，他們自己就是第一批客戶，能做非常快速的回饋。Cowork 的誕生就是因為他們注意到很多非工程師也在用 Claude Code，於是想「我們能不能把這個體驗做得更好？」這個觀察潛在需求的能力，在她過去做過的所有產品中都很管用。
她舉了 Claude for Small Business 的例子。在她幫朋友們導入之後，發現他們常問「有沒有這個 plugin、那個 plugin？」後來 Anthropic 真的推出了 Claude for Small Business，在 Cowork 裡面有一個開關直接整合所有功能。這完全不是她的功勞，但她也注意到了同樣的需求。
她的建議是：永遠保持傾聽回饋、持續迭代、打造令人愉悅且可靠的高品質體驗，同時留意那些你沒預期到的使用場景。當你看到人們費盡心思想讓某個功能運作時，能不能把那個體驗變得更順暢？客戶會用你想不到的方式來使用你的產品，好的或壞的都有，所以最好的方式就是貼近回饋、持續迭代學習。

▍ Claude Routines：非同步工作的新時代
Fiona 說工程的下一個前沿是更多的非同步工作（asynchronous）。Routines 之所以有趣，是因為她以前是同步地下 prompt，然後可能非同步地啟動不同的 prompt。但現在她可以用一個 routine 來幫她產生那些 prompt。抽象層級一直在往上拉。
她用晨間咖啡的例子解釋。以前她每天早上手動翻閱 Slack 回饋頻道，如果有 maker time 就自己去處理一些缺口。現在她設定一個 routine 在每天早上特定時間自動執行，指令大概是「看這個回饋管道，歸納主題，如果發現一些 bug 就產生 polish 修復的 PR」。她醒來時已經有可以 review 的 PR，而不是以前那種看完資訊還要自己想下一步怎麼辦。這是更高層次的抽象，用 routine 來產生 prompt、去啟動不同的 agent。
作為管理者，這意味著你可以把日常要做的事情設定成 routine，例如「專案進度如何？什麼落後了？該支援誰？怎麼改善 polish？」Claude 執行完呈現結果讓你 review。甚至到一定程度你可以給更多自主權，「如果驗證夠好，就直接上吧。」

▍ 從 Token Maxing 到 ROI
Lenny 觀察到最近有一個風向轉變，從「盡量用光所有 token、看看什麼是可能的」轉向「等等，我們到底從中得到了什麼？花了多少錢？ROI 是什麼？」Boris 以前在 Meta 就是負責衡量工程生產力的。
Fiona 說工程生產力的衡量一直是個迷人的話題。一開始看程式碼行數，但有人行數超高是因為搬了一整個 library 進來。那改看「有意義的程式碼行數」？但如果你更新框架之後產出更少的程式碼、功能卻一樣呢？那看 PR 從建立到合併的時間？每個指標都有它的問題。
她的建議是：產出（output）有沒有真正導向成果（outcome）？Token maxing 就像過去的程式碼行數迷思，你衡量的是使用量而不是影響力。她很喜歡一句話：「別把動作誤認為進步。」（Don't forsake motion for progress.）如果你只是在衡量工具使用量，那你衡量的是行動，但那真的在推動最終成果嗎？要拉遠來看你在解決什麼問題、什麼是好的衡量方式，然後聚焦在那裡。
她也強調指標本身需要跟著環境一起演進。她有一個成長心態的原則：你現在用的指標還在服務你嗎？永遠要保持警覺，不要戴著眼罩盲目追隨一個曾經有意義的指標，因為有時候環境變化太快，指標本身也需要調整。
她也建議領導者去做一輪傾聽之旅（listening tour），特別是跟資深工程師聊天，聽他們說什麼有效、什麼沒用、怎麼改善。這些對話常常比儀表板更有啟發。
她分享了 Facebook Marketplace 早期的經驗。他們按區域上線，想要確保在每個區域建立令人滿意的產品體驗後才擴展。原本用「賣家數量」當擴展門檻，但有個區域賣家數量很低，看起來不達標。她仔細一看，發現人們其實在找到他們需要的東西，因為那裡雖然賣家數量少，但有 power sellers。如果盲目按照原來的指標，他們就不會擴展到那個區域，而事實上那個區域運作得很好。所以他們調整了指標，把 power sellers 的因素納入考量。
她的建議是，不管是生產力指標還是產品指標，永遠要保持警覺，確保你不是盲目追隨一個曾經有意義但環境已經變了的指標。人們會用你意想不到的方式使用產品，出貨、觀察、學習、迭代，這個循環永遠不能停。

▍ Bad vs Sad 品質框架
在品質管理上，Fiona 提出了一個「bad vs sad」的框架。Bad 是非常嚴重的不可恢復錯誤。Sad 是痛點但可恢復的問題。有趣的是當你累積太多 sad，它們可能堆疊成 bad。
她讓每個團隊按照自己負責的產品面定義什麼算 bad、什麼算 sad。例如在 CLI 上，crash 就是 bad，因為你的工作全部遺失了。畫面閃爍可能是 sad，可以恢復，但體驗不好。這個框架的好處是，當產品面很多的時候，你不用盯著一堆不同的效能數字試圖判斷「這算好還是不好」，而是有一個統一的方式來俯瞰整體體驗品質。
Lenny 總結說他聽到的重點是，保持品質的最好工具是監控和測試，而不是花更多時間做 review。因為速度太快根本跟不上，不如讓 agent 自己知道成功是什麼樣子，自己去修。Evals 也是這個體系的一部分，加上監控故障率和效能指標，形成一個完整的品質防線。
Fiona 強調要主動出擊做品質管理，找到關鍵體驗的衡量指標，看趨勢變化，盡早偵測品質問題。她說他們團隊在這方面還在持續精進，希望能做得更好。
Lenny 跟 Fiona 確認，重點不在花更多時間 review code（那個速度根本不可能跟上），而是投資在閉環系統上，讓 agent 自己知道成功的定義，在框架內自行解決問題。Fiona 同意這個方向。

▍ 所有管理者先從 IC 做起
Fiona 在 Anthropic 的做法是讓每個新任管理者先從 IC（個人貢獻者）做起。她剛加入時做了一輪傾聽之旅，聽到的回饋包括「我很感謝 agency，但優先順序怎麼定？」以及「review 層數不要太多」。她意識到如果一個主管先用 maker time 深入程式碼、學習 codebase，不用馬上擔心支持人的責任，他們不但能建立對產品的理解，還能跟團隊建立真正的默契。
如果新主管一來就打開管理工具箱做「管理者該做的事」，他們會錯過那個跟團隊建立關係的機會。但如果你先花時間當一個工程師和隊友，大家看到你在用團隊做的產品、給出使用者回饋，那種信任和默契是完全不同的。
Fiona 自己在 Meta 也是一樣，面試進去是管理職，但第一季她也是 IC，因為她想知道身為 Meta 工程師是什麼體驗。她在 Microsoft 是從零開始的，懂所有 codebase、工具和語言，但到 Meta 要重新學。她說 Claude Code 讓她重新體驗了出貨的快感。她上次出貨 production software 大概是 2017 年，之後一直太擔心會搞壞什麼東西。但在 Claude Code 上的第一週，她用 Claude 當 onboarding buddy，問 codebase 的問題，請它幫忙寫自動化測試也做手動測試，慢慢找回了出貨的信心。她有很多當了多年管理者的朋友也因為 Claude 重新開始寫 code 了。

▍ 技能退化：雙重點擊你的依賴
Fiona 說團隊內部確實在討論這個問題。Boris 當初親手寫了早期的程式碼，雖然他現在不再這樣做了，但他從中獲得的知識是無法取代的。所以對每一個加入的工程師，她都會說要花時間理解架構、理解你做的每一個改動，因為這回到 trust but verify。也許有一天這不重要了，但以目前的速度，她仍然認為理解你依賴的那一層是必要的。她把這叫做「雙重點擊你的依賴」（double-click on your dependencies），因為當你的依賴改變時，你才能意識到並利用新的能力。
