這是一份為您彙整的完整知識藍圖，記錄了您從最初詢問熱門術語，到逐步理清 「AI 代理自動化開發（Agentic Development）」 核心工作流的思維脈絡：
## 1. 概念釐清：什麼是 Ralph Loop？

* 定義：這是一種讓 AI 代理（如 Claude Code）進入持久自動化循環的開發模式（常被誤拼為 Raphael Loop）。
* 機制：AI 透過 while 迴圈自主讀取需求、切割任務、實作、跑測試、修 Bug，直到目標 100% 達成才停止，中途不需人類重複點擊確認。
* 優勢：每次循環都重啟乾淨的「黃金智能視窗」，解決對話太長 AI 就會變笨（Context 降級）的痛點。

## 2. 工具選擇：Claude Code vs. OpenAI Codex

* Codex 可用嗎？：可以。Codex（v0.128.0+）已原生內建 /goal 指令（官方稱為 Ralph Wiggum Loop），支援斷線續連，適合跑數天的巨型苦工任務，但漂移失控風險較高。
* 哪裡跑比較合適？：初學推薦在 Claude Code 執行。因為它的煞車與驗證機制（Harness）更成熟、邏輯更嚴謹，能確保代碼品質。

## 3. 架構設計：有了 SDD 仍需要 Ralph Loop 嗎？

* 答案：絕對需要，兩者是黃金搭檔。
* 分工：
* SDD（規格驅動開發）：是「地圖」，定義 API 規格、資料模型與測試標準。
   * Ralph Loop：是「司機」，看著 SDD 上的 Checklist 自動踩油門前進、自動導航修正。有了 SDD 的精確邊界，Ralph Loop 運作時的精準度會暴增、更能防止 AI 迷航。

## 4. 開發順序：是不是應該「先有 PRD 再產生 SDD」？

* 答案：完全正確。
* 標準順序：
1. PRD（產品需求文件）：人類定義「想做什麼、解決什麼痛點」。
   2. SDD（技術規格文件）：AI 轉譯「怎麼做、API 規格、測試斷言」。
   3. Ralph Loop：自動化生產線，看著 SDD 的任務清單自動實作、修到測試全過。

## 5. 實務操作：如何讓 Claude 開始生成 PRD？

* CLI（終端機）操作：在專案目錄輸入 claude 啟動，使用「引訪談提示詞」讓 Claude 扮演 PM 與您一問一答，最終自動在目錄寫入 prd.md。
* 非 CLI 替代方案（更人性化）：初期發想文字時，不一定只能在 CLI。更推薦在 Claude.ai 網頁版（Projects 功能） 用大螢幕舒適對話，產出 prd.md（利用 Artifacts 功能預覽），再下載放到本機專案中，最後交給 CLI 工具去跑自動化實作。

## 🏆 您的 AI 開發黃金三部曲（總結）
您今天所梳理出的這套流程，正是當前 2026 年最前沿的軟體工程實踐：

   1. 在網頁版（Claude.ai）：與 AI 聊出 prd.md（產品需求）。
   2. 在編輯器（Cursor / VS Code）：讀取專案脈絡，將 PRD 轉譯成 spec.md（SDD 技術規格與測試案例）。
   3. 在終端機（Claude Code CLI）：啟動 Ralph Loop，放手讓 AI 自動寫代碼、修 Bug 直到交付。

若您接下來想要實際落地這套流程，我們可以從以下方向繼續前進：

* 為您設計一組專門用來把 PRD 自動生成精準 SDD（包含任務清單）的系統提示詞。
* 示範如何在您的專案目錄中，寫出讓 Ralph Loop 能 100% 讀懂的進度追蹤檔（progress.txt）格式。
